# /nch 累计资金快照整点边界少算 Bug 记录（2026-02-13）

## 现象
- 用户在 `/nchBTC` 里看到的数据与预期不一致。
- 典型表现：整点行（例如 `09:00` / `12:00`）偏小，和“盘间快照”人工复算对不上。
- 输出文案是“累计资金快照 (UTC口径 / 本地展示)”，但用户预期是“盘间资金快照”。

## 如何复现
1. 用旧逻辑运行 `/nchBTC`。
2. 观察整点行，例如 `12:00`。
3. 同时用 ClickHouse 复算：
   - 口径 A（旧逻辑）：累计到 `< 12:00`（实际上只到 11:59）。
   - 口径 B（正确边界）：累计到 `<= 12:00`（包含 12:00 这根 1m）。
4. 对比会发现：机器人输出对齐口径 A，而非口径 B。

## 根因
- 文件：`apps/api-go/internal/telegram/query.go`
- 函数：`intradayNetSnapshot` 内部 `cumAt`。
- 旧代码：
  - `for idx < len(entries) && entries[idx].ts < cutoff { ... }`
- 问题：
  - 整点快照的 cutoff 是 `HH:00`，但条件是 `< cutoff`，导致 `HH:00` 这一分钟被漏掉。

## 修复
- 最小改动 1 行：
  - 从 `< cutoff` 改为 `<= cutoff`。
- 新代码：
  - `for idx < len(entries) && entries[idx].ts <= cutoff { ... }`
- 影响：
  - 整点行改为“包含整点分钟”，与盘间快照直觉一致。

## 验证
- 代码自检：`go test ./...`（`apps/api-go`）通过。
- 数据对账：
  - 以 BTCUSDT 为例，用 ClickHouse 按 `<= HH:00` 复算后，与修复后的整点行对齐。
- 运行态注意：
  - 仅改代码不够，必须重建并重启 `api` 容器；否则 TG 仍返回老逻辑结果。

## 附：这次容易误判的点
- 用户看到“没变化”并不一定是修复无效，常见是容器没更新到新二进制。
- 当前时刻行（如 `HH:41`）受进行中分钟影响，数值可能抖动；整点行最适合做边界校验。

## 后续建议
1. `/nch` 文案改成“盘间资金快照”，避免“累计”歧义。
2. 明确当前行口径（是否只算已闭合分钟）。
3. 给 `/nch` 增加一个最小单测：构造整点边界数据，断言不会漏 `HH:00`。
